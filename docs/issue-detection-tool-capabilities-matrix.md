# Issue-Detection Tool-Capabilities Matrix

**Why this exists.** Before investing more effort planting items, assess which of our checklist error
classes are *already* caught by off-the-shelf linters / static analyzers. This matters two ways:

1. **Experiment validity** — if a deterministic linter solves an item *completely*, an LLM-grouping
   study on that item is low-value (you'd just run the linter). The interesting items for grouping are
   the ones static analysis can't fully catch — semantic/judgement smells.
2. **Realism** — cal.com already runs ESLint + `@typescript-eslint/recommended` + a custom
   `@calcom/eslint` plugin + Prettier (verified in `packages/config/eslint-preset.js`). A planted issue
   that the repo's own lint would flag is unrealistic (it wouldn't survive CI).

> **Generated content.** The scale table, both capability tables, the tool-capability-difference notes, and the per-item
> notes below are **generated from [`data/issue-detection-tool-capabilities-matrix.json`](../data/issue-detection-tool-capabilities-matrix.json)** by
> [`scripts/issue_detection_tool_capabilities_matrix_report.py`](../scripts/issue_detection_tool_capabilities_matrix_report.py). Edit the JSON (not the tables), then run
> `scripts/issue_detection_tool_capabilities_matrix_report.py` to rewrite the regions between the `<!-- GENERATED -->` markers. The
> pre-commit hook runs `--check` and fails if the doc is stale vs the data. Prose outside the markers is
> hand-written and safe to edit directly.

## Scale (ordinal)

Cells render the ordinal as an icon; **a blank cell means the language's tool has no detector for that
item** (kept blank for readability rather than writing a `0`/"None").

<!-- BEGIN GENERATED: scale -->
| Cell    | Ordinal      | Meaning                                                                                                                                       |
| ------- | ------------ | --------------------------------------------------------------------------------------------------------------------------------------------- |
| ✓       | 2 — Complete | A standard rule/analyzer detects essentially all instances deterministically (mechanical pattern).                                            |
| ∙       | 1 — Partial  | A rule catches some special cases / a syntactic proxy, but misses the general semantic form (false negatives), or needs opt-in config.        |
| (blank) | 0 — None     | No off-the-shelf detector; requires semantic/contextual judgement (LLM or human).                                                             |
| ?       | ?            | Not yet researched — this item/tool cell has not been investigated. A research-worklist placeholder, distinct from a determined blank or n/a. |
<!-- END GENERATED: scale -->

- **mypy / `tsc`** are *type-checkers*, not lint rule engines: they catch unused code (`tsc` `noUnusedLocals`) and unsafe-`any` symptoms but have no empty-catch / metric / magic-number / clone rules → blank on those rows.

- **SpotBugs** is bug-pattern (bytecode) focused: it lacks a magic-number rule and a CPD clone pass (those live in **PMD** / PMD-CPD), so 16.7.6 and 16.4.1 are PMD-only on the Java side.
- **Biome** ships a smaller rule set than ESLint/SonarJS: it has `noEmptyBlockStatements`, `noDoubleEquals`, `noVar` (✓ there) but no metric/magic-number/CPD rules → blank on those.
- **golangci-lint** bundles many linters (incl. `gocyclo`/`funlen`/`dupl`) so it carries the metric + SRP coverage that bare **staticcheck** doesn't; both catch the unused/dead-code case.
